# 新見積もり計算ツール 設計見積（初版）

## 1. 背景と目的

既存の配布済み見積もり自動作成ツールはロックされており、利用側で直接改善できない。現状リポジトリには VBA マクロと `書式テンプレ.xlsx` があり、既存 DB を活用しながら新しい見積もり計算ツールを作成する方針とする。

本ドキュメントでは、実装に入る前の初期設計と概算工数を整理する。計算式は現状資産に含まれるため、承認前には計算ロジックを変更しない前提で見積もる。

## 2. 現状資産の整理

| 資産 | 内容 | 初期方針 |
| --- | --- | --- |
| `見積書のVBA/` | 既存 Excel VBA の抽出コード。基本設定読み込み、DB参照、見積書への転記、各種クリア処理などが含まれる。 | 仕様確認用の参考実装として扱い、必要な入出力・帳票項目・既存計算式の把握に使う。 |
| `書式テンプレ.xlsx` | 見積書出力用テンプレート。現在は「見積書」シートを持つ。 | 出力レイアウトのたたき台として利用する。新ツールではページ別シート生成を検討する。 |
| 既存 DB | 商品・材料・工種・単価などのマスタを保持している想定。 | DB スキーマと接続方式を確認し、読み取り中心で連携する。更新が必要な場合は別途承認する。 |

## 3. 前提・制約

- 既存計算式・金額算出ルールは、ユーザー承認なしに変更しない。
- DB は既に存在するため、初期開発では DB の再設計を行わず、既存スキーマを活かす。
- 既存ツールを編集できないため、新ツールは独立した成果物として作成する。
- 見積書は Excel ファイルとして出力できることを優先する。
- 従来は 1 シート内で見積書を縦方向に展開していたが、新ツールではページごとにシートを分け、シート名には部位名（既存メモでは D12）を利用する。
- 既存 Excel 利用者が多い前提で、入力・確認・出力の操作負荷を下げる。

## 4. 推奨アーキテクチャ

### 4.1 推奨案：Python 製デスクトップ/簡易 Web ツール + Excel 出力

| レイヤー | 役割 | 候補技術 |
| --- | --- | --- |
| UI | 案件情報、部位、仕様、数量などを入力・選択する画面 | Streamlit / NiceGUI / PySide |
| アプリケーション | 入力検証、DB 参照、見積データ構築、出力指示 | Python |
| DB 接続 | 既存 DB からマスタ・単価・工種情報を取得 | DB 種別に応じた Python ドライバ |
| 計算 | 既存計算式を移植または Excel テンプレート式を利用 | 初期は Excel 式維持を優先 |
| 帳票出力 | ページ別シートを作成し、テンプレートに値を転記 | openpyxl / xlwings |

#### 推奨理由

- VBA ロック問題から切り離して開発できる。
- DB 接続・入力チェック・ログ出力を実装しやすい。
- Excel 出力を維持できるため、利用者の運用変更を最小化できる。
- 将来的に Web 化・権限管理・履歴管理へ拡張しやすい。

### 4.2 代替案

| 案 | メリット | デメリット | 評価 |
| --- | --- | --- | --- |
| Excel VBA 新規作成 | 既存運用に近い。Excel だけで完結しやすい。 | 保守性・配布・ロック・バージョン管理の課題が残りやすい。 | 小規模なら可。長期運用では非推奨。 |
| Access / FileMaker | DB 画面を作りやすい。 | ライセンス・実行環境依存が出る。Excel 帳票連携の作り込みが必要。 | 社内標準があれば検討。 |
| Web アプリ本格構築 | 複数人利用、権限管理、履歴管理に強い。 | 初期費用・サーバ運用が増える。 | 中長期の本命。ただし初期版には過剰な可能性。 |

## 5. 想定機能スコープ

### 5.1 初期版（MVP）

1. 案件基本情報の入力
   - 顧客名、案件名、見積日、担当者、備考など。
2. 部位・ページ情報の入力
   - ページ番号、部位名、仕様番号、数量、単位など。
3. DB マスタ検索・選択
   - 工種、材料、単価などを既存 DB から参照。
4. 入力チェック
   - 必須項目、数値範囲、DB 未登録、重複、空行など。
5. 見積計算
   - 初期は既存計算式と同等になるように実装・検証する。
6. Excel 出力
   - `書式テンプレ.xlsx` をベースに、ページごとのシートを生成する。
   - シート名は部位名を基本とし、重複・禁止文字・31文字制限に対応する。
7. 検算用ログ出力
   - 入力値、DB 参照結果、計算に使った係数、出力先を記録する。

### 5.2 初期版では対象外

- 既存 DB の大幅なスキーマ変更。
- 単価改定ワークフロー。
- 見積承認フロー。
- 複数ユーザー同時編集。
- 既存計算式の業務的な見直し。

## 6. データフロー案

```text
[利用者入力]
    ↓
[入力チェック]
    ↓
[既存 DB 参照]
    ↓
[見積データ生成]
    ↓
[既存計算式相当の計算 / Excel 式反映]
    ↓
[ページ別シート生成]
    ↓
[Excel 見積書出力]
    ↓
[検算・操作ログ保存]
```

## 7. DB 確認事項

設計確定前に、次の DB 情報が必要となる。

| 確認項目 | 必要な理由 |
| --- | --- |
| DB 種別（Access、SQL Server、MySQL、SQLite、Excel など） | 接続方式・配布方式・必要ライブラリが変わる。 |
| テーブル一覧・カラム定義 | 画面項目と帳票項目のマッピングに必要。 |
| 主キー・外部キー・コード体系 | マスタ選択、参照整合性、重複検出に必要。 |
| 単価・係数の有効期間 | 見積日時点の金額再現性に必要。 |
| 読み取り専用か更新ありか | 権限・トランザクション・監査ログ設計に影響する。 |
| 既存ツールとの参照差分 | 新旧ツールの計算結果一致を検証するため。 |

## 8. 画面設計ラフ

### 8.1 入力画面

- 案件基本情報エリア。
- 部位・ページ一覧エリア。
- 明細入力グリッド。
- DB 検索パネル。
- エラー・警告表示エリア。
- Excel 出力ボタン。

### 8.2 出力 Excel

- 1 ページ = 1 シート。
- シート名 = 部位名。ただし Excel 制約に合わせて自動補正する。
- テンプレートの書式・罫線・計算式は可能な限り維持する。
- 出力後に人手で確認・微修正できる形式にする。

## 9. 概算工数

DB スキーマと計算式の確定度により変動するため、初期見積は幅を持たせる。

| フェーズ | 主な作業 | 目安工数 |
| --- | --- | ---: |
| 1. 要件確認 | 既存 VBA・テンプレート・DB・現行運用の確認、出力サンプル収集 | 2〜4 人日 |
| 2. 基本設計 | 画面、データ構造、DB 参照、Excel 出力、ログ方針の設計 | 3〜5 人日 |
| 3. プロトタイプ | DB 接続、入力画面、1ページ分の Excel 出力、簡易検算 | 5〜8 人日 |
| 4. MVP 実装 | 複数ページ対応、入力検証、テンプレート反映、エラー処理 | 8〜15 人日 |
| 5. 計算結果照合 | 既存ツールとの結果比較、差分調査、承認反映 | 5〜10 人日 |
| 6. 利用者確認 | 操作確認、要望反映、帳票微調整 | 3〜7 人日 |
| 7. 配布準備 | 実行手順、設定ファイル、簡易マニュアル、バックアップ手順 | 2〜4 人日 |
| **合計** | 初期版リリースまで | **28〜53 人日** |

### 9.1 期間目安

- 1 名体制: 約 1.5〜3 か月。
- 2 名体制: 約 1〜2 か月。
- DB 情報・計算式・出力サンプルが早期に揃うほど短縮可能。

## 10. リスクと対策

| リスク | 影響 | 対策 |
| --- | --- | --- |
| 既存計算式の仕様が VBA/Excel 内に分散している | 新旧結果の差分が出やすい。 | 先に計算項目一覧と検算ケースを作る。 |
| DB のコード体系が不明確 | マスタ選択ミスや単価不一致が起きる。 | DB 定義書または実データから参照ルールを整理する。 |
| Excel テンプレートの帳票制約 | ページ追加やシート名生成で崩れる可能性。 | テンプレート複製方式で実装し、出力後比較を自動化する。 |
| 現場ごとの例外入力 | UI が複雑化する。 | MVP では共通ケースを優先し、例外は備考・手入力欄で逃がす。 |
| 配布先 PC の環境差 | 起動不可・DB 接続不可が起きる。 | Python 実行ファイル化または社内標準環境に合わせる。 |

## 11. 次に決めること

1. DB 種別と接続方法。
2. 既存 DB のテーブル定義またはサンプルデータ提供範囲。
3. 既存ツールで作成した正解見積書サンプル 3〜5 件。
4. 新ツールの実行形態。
   - 個人 PC で起動するデスクトップ/簡易 Web。
   - 社内サーバで動かす Web アプリ。
   - Excel アドイン/マクロブック。
5. 初期版で必ず対応する帳票パターン。
6. 計算ロジックを Excel 式として残すか、アプリ側へ移植するか。

## 12. 初期結論

初期版は「Python 製の入力・DB参照ツール + Excel テンプレート出力」で進めるのが最もバランスがよい。既存 VBA は仕様把握と検算に使い、計算式は承認があるまで変更しない。まず DB 定義と正解サンプルを確認し、1ページ分の出力プロトタイプを作成してから、ページ別シート生成と入力チェックを拡張する。
